Project Context

Why I built this

I built Sift around two problems I kept running into.

The first was saved media. I would save Instagram reels, posts, carousels, and YouTube videos but later found it hard and laborious to scroll through and search for them. I wanted one pipeline that could take these sources as inputs, process them, and present them to me all together so that I could access the information at a later time.

The second was music. I already had an extensive collection spread across Spotify, playlists, YouTube links, track names, and other sources. What I wanted was my own offline library built on top of that existing collection, without manual search and download. One that I could access anywhere, anytime.

Sift brings both workflows into one task-based system: submit the source, let the backend process it, review anything uncertain, and keep the final output organized.

Lane-based task runtime

Music downloads, Instagram extraction, OCR, and transcription can take very different amounts of time. Sift runs them through separate task lanes, tracks queue position, and keeps the UI updated without blocking the rest of the app.

Task lifecycle tracking

Each run carries its own status, metadata, artifacts, errors, review items, and completion state. That lets Sift handle jobs that move through queueing, processing, review, failure, and final output without treating them as simple one-shot requests.

Mixed-content pipeline design

Sift handles inputs that can contain images, videos, audio, captions, and carousel items inside the same run. The backend separates each part into the right processing path and then assembles the outputs back into one result.
